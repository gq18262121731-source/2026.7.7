# Coding Rules

本文件专门约束 AI 编程行为。

## 修改范围

不允许修改无关文件。

每次任务只完成一个独立功能或一个明确修复。

如果必须修改共享模块，必须确认影响范围并保持兼容。

## 复用优先

优先复用已有：

- Layout
- Components
- Hooks
- Services
- Types
- Theme tokens

不要重复造轮子。

## 类型规则

保持 TypeScript 严格类型检查。

禁止随意使用 `any`。

需要临时兼容时，应使用明确类型、unknown、类型守卫或 schema。

## 数据规则

所有页面必须通过 API 获取业务数据。

即使是 Mock 数据，也要走 API 或明确 service 层。

禁止把 Mock 数据写死在页面组件中。

## 组件规则

组件职责必须单一。

避免：

- 超大组件
- 页面中混入复杂业务逻辑
- 重复状态管理
- 深层 prop drilling

## 状态规则

新增功能必须包含必要的：

- Loading
- Error
- Empty
- Success

不能只处理成功路径。

## 技术栈规则

不引入未经批准的新技术栈。

如需新增依赖，必须满足：

- 对项目有明确收益
- 不明显增加复杂度
- 与现有技术栈兼容

## 可扩展规则

所有代码应以未来替换真实能力为前提：

- YOLO
- 数据库
- LLM
- RAG
- Make

替换真实能力时，前端页面原则上不应大改。

## 验证规则

开发完成后必须尽可能运行：

- 类型检查
- lint
- 单元测试或 smoke test
- 前后端启动验证

如果无法运行，必须说明原因。
